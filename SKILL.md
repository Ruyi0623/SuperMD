# SuperMD

Generate, convert, audit, improve, and transform Markdown documents.

## 一、核心能力

五种模式。先判断用户需要哪个，不明确时一句话确认。

### Generate — 从零生成
根据描述创建结构化 Markdown 文档。识别文档类型和受众，匹配模板，填入内容。

### Convert — 原始内容转 Markdown
将非结构化文本（粘贴的笔记、HTML 片段、松散格式内容）整理为规范 Markdown。从内容本身推断结构——发现自然的段落分割、重复模式、属于表格的数据。保留所有信息，移除格式噪音。无法推断的内容用 `<!-- Missing: ... -->` 标注，不凭空编造。

### Audit — 审查诊断
逐项检查 `.md` 文件，输出三级诊断报告（🔴 严重 → 🟡 建议 → 🟢 优点）。只诊断不修改，让用户了解问题全貌后再决定。

### Improve — 优化改进
对现有 `.md` 文件进行结构优化和排版美化。与 Audit 不同：Improve 直接产出优化后的版本，不输出诊断报告。与 Convert 不同：输入已是 Markdown，不是原始文本。

触发词："优化这份文档"、"改善结构"、"让它更好看"、"排版优化"、"polish this doc"、"clean up this markdown"。

优化原则：
- 不改变内容含义，只改善结构和格式
- 修复所有结构问题（层级跳跃、裸 URL、无语言标签的代码块）
- 统一列表和代码块风格
- 重新组织段落，拆分过长段落（>7 行），合并零碎段落
- 修正中文排版（中英文空格、标点符号）
- 如有明显的信息组织问题（重复内容、逻辑顺序混乱），优化后简要说明改动

### Switch — 格式转换
Markdown 风格互转：

| 源格式 | 目标格式 | 关键变更 |
|--------|----------|----------|
| GFM | Obsidian | `[text](url.md)` → `[[url\|text]]`；callout 转小写 |
| Obsidian | GFM | `[[wikilinks]]` → `[text](url)`；`#tags` 保留 |
| GFM | MDX | 代码块添加 `live` 属性；补充 import 语句 |
| Standard | GFM | 启用自动链接、任务列表、表格、删除线 |

转换 Obsidian wikilink 时，先确认对应 `.md` 文件是否存在以正确解析路径。

## 二、工作流程

1. **识别模式** — 生成、转换、审查、优化还是转换格式？
2. **必要时澄清** — 缺少关键信息（文件路径、目标风格、受众）时简短提问
3. **执行** — 按本规范的模式对应章节处理
4. **输出** — 遵循输出规范（第六节）
5. **保存确认** — 输出后询问是否写入文件，确认路径后保存

## 三、文档结构设计原则

### 3.1 层级规范
- `# H1`：文档标题，**每个文档只出现一次**
- `## H2`：主要章节
- `### H3`：子章节
- `#### H4`：仅在必要时使用，避免层级过深
- 层级不可跳跃（H1 → H3 是错误的）

### 3.2 内容块设计
- **代码块**：始终标注语言标识符，使用具体语言而非 `code`
- **表格**：列标题简洁，对齐方式有意义（数字右对齐，文字左对齐）
- **列表**：有序列表用于步骤/排名，无序列表用于并列项
- **引用块**：用于重要提示、警告、引用原文
- **分隔线 `---`**：在语义上划分重要章节，不滥用

### 3.3 可读性规范
- 段落之间保留空行
- 列表项之间如果内容较长，加空行使其更易读
- 行内代码：所有命令、变量名、文件名、函数名都用反引号包裹
- 链接：使用描述性文字，避免裸 URL（除非刻意展示）
- 图片：必须有 alt 文字：`![描述性文字](url)`
- 中文排版：中英文/数字间加空格；中文语境使用中文标点；代码和 URL 除外

## 四、各类文档的最佳结构模板

### README / 项目文档
```
# 项目名称
简短描述（1-2 句）

## ✨ 特性
## 🚀 快速开始
### 安装
### 使用示例
## 📖 文档
## 🤝 贡献
## 📄 许可证
```

### 技术教程 / 博客
```
# 标题（吸引人，包含关键词）
> 摘要：本文将...（1-2 句）

## 前提条件
## 背景知识（可选）
## 步骤一：...
## 步骤二：...
## 常见问题
## 总结
## 参考资料
```

### 学习笔记
```
---
tags: [subject, topic]
date: YYYY-MM-DD
source:
---

# 主题

## 核心概念
## 关键点
## 示例
## 个人理解/问题
## 相关链接
```

### API 文档
```
# API 名称

## 概述
## 认证
## 端点列表
### GET /resource
**参数** | **响应** | **示例**
## 错误码
## 更新日志
```

### 会议记录 / 周报
```
# 会议/周报 — YYYY-MM-DD

## 参与者
## 议题 / 本周工作
## 决定事项
## 待办 Action Items
- [ ] 任务 @负责人 Due: 日期
## 下次议题
```

## 五、Markdown 审查清单（Audit Mode）

逐项检查并输出诊断报告。

**结构问题**
- [ ] 是否有且只有一个 H1 标题
- [ ] 标题层级是否连续（无跳跃）
- [ ] 文档是否有清晰的开头、主体、结尾

**格式问题**
- [ ] 代码块是否都有语言标识
- [ ] 列表格式是否统一（`-` 或 `*`，不混用）
- [ ] 链接是否有描述性文字
- [ ] 图片是否有 alt 文字

**可读性问题**
- [ ] 段落是否过长（建议单段落不超过 5-7 行）
- [ ] 是否有不必要的粗体/斜体滥用
- [ ] 表格是否有对齐说明

输出格式：
```
## 审查报告

### 🔴 严重问题（影响渲染）
### 🟡 建议改进（影响可读性）
### 🟢 优点（做得好的地方）
### 📝 优化后的版本
```

## 六、输出规范

1. **直接输出 Markdown 源码**，不加额外解释（除非用户要求说明）
2. 如果文档较长，先给出**结构大纲**，确认后再展开写
3. 输出时使用**代码块包裹**完整的 Markdown 内容，方便用户复制
4. 如有多个版本或风格选择，给出最多 2 个版本供选择
5. 转换任务：保留原始内容的核心信息，不添加无中生有的内容
6. 优化任务：不改变内容含义，只改变结构和格式

## 七、沟通风格

- 简洁直接，减少冗余说明
- 对于大型文档（>500 词），先确认结构再写，避免返工
- 用户如果说"直接写"或"just do it"，跳过确认，立即输出
- 遇到歧义时，做出合理假设并说明，不反复追问
